How it went

FLASK

The first step was to understand how to build a website using python, after doing some googling two frameworks kept showing up flask and Django. I ended up choosing flask because it seemed  straightforward.

Went through official flask docs but stack overflow seemed to be easier to get the information I needed to get started. I ran pip  install flask and created my first route! That pointed to a HTML file with hello world text üòÉ

I wanted to separate the website content from the HTML, so I had an idea to put the website content in JSON, then read that JSON file and pass the content to the flask render_template function, and finally use jinja2 to render the content.

Creating a JSON file wasn't as hard once I realized it was a list of dictionaries. Hence I could loop through and use an index to access different entries.

SOLVING BUGS

Once I had created my website I started to realize certain bugs when I tested it especially when I tried to access my site on small screen devices. Some problems were:

  • navbar overlapped onto text when clicked
  • button sizes appeared smaller
  • the footer was fixed and hence hid the page
  • headers appeared smaller

To overcome some of these challenges: my javascript initiation beganüòÖ. I used js to add event listeners to the Navbar and to learn this

While doing some performance tests, I noticed that flask was not handling many connections per second and reporting a lot of errors! wtf!. I needed  a 'Hermes flask' . I did a bit of research and stumbled on Gunicorn. Gunicorn works by internally calling your flask code, this results in your app being able to handle many requests per second.

DATABASES

I wanted to learn about databases as I came across them in all of the data science platforms. I decided to use the PostgreSQL database add on service that was available on Heroku to store and retrieve my data as opposed to the JSON file that I was using.  I also  needed a faster way to deploy new tests and applications anywhere  rather than doing it on-site on my local computer. I stumbled on Heroku. Heroku is a Cloud Platform as a service (PaaS).

Being quite new at this I initially used a GUI, pgAdmin 4 it made it easier to manipulate schema and data in the database. However, with time i was able to write queries by myself as i progressed.

Next,to connect my python to the database, I imported the psycopg2 library. I inserted dummy data into my DB to test if I could retrieve data from DB using python, after a few syntax errors lol I was successful. I set up a database in Heroku and used the database URL from Heroku to connect and update my HTML files to access information from my  database. ( I always tested this on my local environment before pushing to development). 

At this point the web page looked pretty solid, however going through my code I could see repetitions, so to have cleaner code I used DRY coding principles (Don't repeat yourself). The benefits I saw from this were:

  • maintainability- changing and updating code was way easier
  • reuse - I was able to reuse some parts of the code and this in the long run reduced development time.

MICROSERVICES

Finally, I wanted to dynamically access and manage my data from my browser. I decided to use Forest admin for this. Forest admin provides  an API hosted on your server to retrieve your data. I also used Amazon s3 to store my image files. Amazon s3 is an object storage service that stores objects in buckets.  once this was done I added a file upload action from forest admin that pushes images to s3.

Things learnt:

  • creating s3 bucket policy
  • securing Amazon s3 buckets
  • linking microservices
  • initializing forest admin
  • connecting forest admin to my pgSQL database
